Build Sequence — What comes first?

Three approaches to introducing Clerk + Supabase into Loci. Which order feels right?

A

Auth-first Recommended

Clerk sign-in/sign-out → Supabase schema + RLS → EPUB upload to Storage → Data sync (annotations, prefs)

Why

  • Books are owned by users — you can't store books without knowing who owns them
  • Clerk is one integration that unlocks everything; do it once, never touch auth again
  • Anonymous users see "Sign in to save your library" — clean progression

Trade-off

  • Users can't upload books until auth is live — may feel slow to ship
B

Books-first (anonymous)

EPUB upload to Supabase Storage anonymously → add Clerk auth → migrate anonymous books to user account on sign-in

Why

  • Core feature (book library) ships sooner without auth friction
  • Users can try it before signing up

Trade-off

  • Requires migration logic when user later signs in
  • Anonymous storage = no user isolation, harder to secure
  • More complex data model to reconcile anonymous → authenticated state
C

Auth + Books together

Build Clerk integration and basic book upload in the same sprint — no reading experience until signed in

Why

  • No migration complexity — books always owned by a user from day one
  • Simpler data model overall

Trade-off

  • Longer until anything ships — auth + books + storage in one go
  • More context to hold at once during implementation
